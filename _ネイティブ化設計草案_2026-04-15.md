# tools/ ネイティブ HTML 化 — 設計草案

**作成日:** 2026-04-15
**作成者:** Claude（コード整理・モダン化作業の延長として）
**目的:** 現在 iframe + GAS HtmlService で作られている 3 つのツールを、ポータル内のネイティブ HTML に移行する計画を紙の上で固める。実装はまだしない。

---

## 1. なぜやるのか

| 観点 | 現状 (iframe) | ネイティブ化後 |
|---|---|---|
| **ダークモード** | ❌ iframe 中身は白いまま | ✅ 外側と同じテーマ |
| **初回表示速度** | iframe を別 URL から再取得 | ポータル内で即描画 |
| **オフライン** | GAS が叩けない = 全滅 | Service Worker でキャッシュ対応可能 |
| **スタイル統一** | tools/ は Tailwind、外側は CSS 変数 | 全部 CSS 変数で揃う |
| **iOS PWA の挙動** | iframe 内での戻る・コピー等がぎこちない | ネイティブ動作 |
| **保守性** | HTML が GAS 内にテンプレートとして埋め込まれている | コードが通常の HTML/JS ファイル |
| **GAS 側の役割** | HTML を返す + ロジック + Notion API | **API だけ**（データ入出力に専念） |

一言でいうと「GAS は DB アクセサに専念させ、画面はポータル内で作る」。これで GAS の変更範囲を最小にしつつ、UI は自由に扱えるようになる。

## 2. 現状のツール棚卸し

| ツール | 現状 | GAS 側の関数 | ネイティブ化規模 |
|---|---|---|---|
| **検査依頼** `tools/inspection.html` | ✅ **既にネイティブ** (1797 行) | `GAS_CONFIG` 経由で GAS fetch | 不要 |
| **新規物件登録** `tools/property.html` | ❌ iframe | `gas-dashboard-data.gs` → `handlePortalSubmit` (既にポータル対応済) | **小〜中** |
| **支払い管理** `tools/payment.html` | ❌ iframe | `gas-payment-management.gs` → 14 関数 (CRUD) | **中〜大** |
| **工程作成** `tools/schedule.html` | ❌ iframe | `gas-schedule-creation.js` → `getFormHtml()` (798 行の HTML 生成) | **大** |

### 2.1 各ツールの複雑度評価

#### 新規物件登録 (property) — ★☆☆☆☆
- **入力:** 物件名・市町村・本体着工・建て方・引渡し + 8 個のチェックボックス
- **出力:** Notion に物件ページを 1 件作成
- **GAS 側 API:** `handlePortalSubmit` **既に実装済み**（ポータル向けのJSON返しエンドポイント）
- **状態:** **フォームを HTML で作り直すだけ**でほぼ完成。既存 iframe 版がそのまま雛形になる

#### 支払い管理 (payment) — ★★★☆☆
- **入力:** 物件選択 → 業者選択 → 金額 → 日付 → 支払い区分 → 備考
- **機能:** 新規登録、既存検索、更新、業者マスタ追加、担当者マスタ追加、一覧表示
- **GAS 側 API:** 14 関数あり（`getProperties`, `getVendors`, `searchPayments`, `submitPayment`, `updatePayment` ...）
- **状態:** API は既にほぼ JSON を返せる構造。ただし **現状は `google.script.run` で呼ばれている前提**なので、doGet/doPost で受けられるようにラップする改修が必要

#### 工程作成 (schedule) — ★★★★☆
- **入力:** 物件名・場所・本体着工・建て方・引渡し日 → ボタン押下
- **機能:** 日付から工程を自動計算（三隣亡・六曜・水日休日回避等の複雑なロジック 200 行超）→ 確認画面 → Google カレンダーに登録 → 確認メール送信 → 完了画面
- **GAS 側 API:** `getFormHtml()` が 600 行超の HTML テンプレを返している
- **状態:** ロジック（日付計算部分）は GAS に残すのが自然。フロント側は **入力 + 結果確認 + カレンダー登録ボタン** の3画面に分解

## 3. 共通アーキテクチャ（全ツール共通）

### 3.1 責務の分離

```
┌─────────────────────────┐       ┌──────────────────────────┐
│ tools/*.html            │       │ GAS (gas-*.gs/js)        │
│ (ポータル内ネイティブ)   │       │ (API 専用エンドポイント)  │
│                         │       │                          │
│ - HTML フォーム描画     │◄─────►│ - Notion API アクセス     │
│ - 入力バリデーション    │ fetch │ - Google Calendar アクセス│
│ - モーダル/トースト     │ JSON  │ - データ加工・日付計算     │
│ - fetch で API 呼び出し │       │ - 結果を JSON で返す      │
└─────────────────────────┘       └──────────────────────────┘
```

### 3.2 GAS 側の API 統一パターン

現状 `inspection.html` がすでにこのパターンで動いているので、それを踏襲する:

```javascript
// フロント側 (tools/xxxx.html)
const GAS_URL = 'https://script.google.com/macros/s/AKfycbxXXXX/exec';

async function callGas(mode, params) {
  const url = GAS_URL + '?mode=' + mode;
  const res = await fetch(url, {
    method: 'POST',
    body: JSON.stringify(params),
    redirect: 'follow',
  });
  return res.json();
}
```

```javascript
// GAS 側 (gas-xxxx.gs)
function doGet(e) {
  // 下位互換: 古い iframe からのアクセスは従来通り HTML を返す
  if (!e.parameter.mode) {
    return HtmlService.createHtmlOutput('このエンドポイントは API 専用です');
  }
  return handleApiRequest_(e);
}

function doPost(e) {
  return handleApiRequest_(e);
}

function handleApiRequest_(e) {
  const mode = e.parameter.mode;
  const params = e.postData ? JSON.parse(e.postData.contents) : e.parameter;

  let result;
  try {
    switch (mode) {
      case 'getProperties':  result = getProperties();     break;
      case 'getVendors':     result = getVendors();        break;
      case 'submitPayment':  result = submitPayment(params); break;
      // ...
      default: throw new Error('Unknown mode: ' + mode);
    }
    return jsonResponse_({ success: true, data: result });
  } catch (err) {
    return jsonResponse_({ success: false, error: err.message });
  }
}

function jsonResponse_(obj) {
  return ContentService.createTextOutput(JSON.stringify(obj))
    .setMimeType(ContentService.MimeType.JSON);
}
```

## 4. 実装順序の推奨

### Phase 1: **新規物件登録** (property.html) — 最初のトライアル
**所要時間の目安: 半日（実作業 2〜3 時間）**

#### なぜ最初か
- **既に GAS 側に `handlePortalSubmit` が実装済み**（ポータル版の API）
- フォームがシンプル（物件名 + 日付 3 つ + チェックボックス 8 つ）
- 本番影響が小さい（1 件追加するだけ、既存データを触らない）
- 失敗しても **iframe 版に戻すだけで復旧**

#### 作業手順
1. **バックアップ:** 現在の `tools/property.html` を `_archive/` に退避
2. **新しい property.html を作成:**
   - 既存 iframe 版の見た目をそのまま HTML で再現（既に Notion 物件登録の UI がある）
   - `#app` の中にフォーム要素を配置
   - CSS は `_tool-common.css` + ツール固有スタイル
3. **送信処理:**
   ```javascript
   document.getElementById('submitBtn').addEventListener('click', async () => {
     const data = collectFormData();
     const params = new URLSearchParams({ mode: 'submitSchedule', ...data });
     const res = await fetch(GAS_URL + '?' + params);
     const json = await res.json();
     if (json.success) showToast('登録完了: ' + json.name);
     else showToast('エラー: ' + json.message, 'error');
   });
   ```
4. **テスト:**
   - ライトモード / ダークモード 両方で表示確認
   - 実際に 1 件テスト物件を作って Notion に反映されるか確認
   - 作ったテスト物件は Notion 側で手動削除
5. **GAS 側の変更は不要**（`handlePortalSubmit` が既にある）

#### 検証ポイント
- [ ] フォームの見た目が既存と同等か
- [ ] ダークモードでも読めるか
- [ ] 物件が Notion に作成されるか
- [ ] エラー時のメッセージ表示が動くか

### Phase 2: **支払い管理** (payment.html) — Phase 1 が安定したら
**所要時間の目安: 1〜1.5 日**

#### GAS 側の改修が必要
- 現状は `google.script.run.getProperties()` 形式で呼ばれている前提
- これを `doGet(e)` + `mode` パラメータで受けられるようにラップ（3.2 のパターン）
- 影響を受ける関数: 14 個（ただしロジック本体は触らず、エントリーポイントを追加するだけ）

#### フロント側の設計
- 画面 1: 物件選択 → 一覧表示（タブ切替で 新規/検索/一覧）
- 画面 2: 新規登録フォーム
- 画面 3: 既存支払い編集モーダル
- 画面 4: 業者マスタ追加（サブモーダル）

#### リスク
- **本番データを書き換える機能**なので、テスト環境（テスト用物件）での動作確認必須
- 更新機能のバグは「金額が間違って登録される」など業務に直結
- 最悪のケース: Notion の「支払い管理」DB が壊れる → Notion の履歴から復元可能だが手間

#### 作業手順
1. GAS 側: `doGet` + `doPost` で mode パラメータを受け取るアダプターを追加
2. GAS 側: テスト用にテスト物件を作成
3. フロント: 読み取り系機能だけ先に実装（物件一覧 / 支払い検索）
4. フロント: 新規登録を実装
5. フロント: 更新機能を実装
6. 実業務で 1〜2 週間並行運用（iframe 版も残して比較）
7. 安定確認後に iframe 版を削除

---

### Phase 3: **工程作成** (schedule.html) — 最難関
**所要時間の目安: 2〜3 日**

#### なぜ難しいか
- `gas-schedule-creation.js` の日付計算ロジックが複雑（六曜・三隣亡・水日休日回避・近い土日検索など 200 行超）
- カレンダーへの登録 + 確認メール送信 + 物件名キャッシュ（スプレッドシート）+ キャンセルトークンまである
- **3 画面構成**: 入力 → 計算結果確認 → 最終登録

#### 設計方針
- **日付計算ロジックは GAS 側に残す**（移植すると六曜判定などが面倒）
- フロント側は「GAS に計算を依頼 → 結果を表示 → OK なら登録指示」の流れ
- 新しい GAS API:
  - `mode=calcSchedules` → 日付配列を返す（実カレンダーには書かない）
  - `mode=registerSchedules` → 確認した日付でカレンダー登録＋メール送信

#### フロント側の設計
- 画面 1: 物件名・場所・着工/建て方/引渡し の 3 日付入力
- 画面 2: 計算された工程一覧（仮数値、まだカレンダーには書かれていない）→ 編集可能
- 画面 3: 登録完了メッセージ

#### リスク
- カレンダー登録は業務のコアなので、失敗すると予定がずれて現場に混乱
- 日付編集できるようにすると UI が複雑化
- **最初のバージョンは「計算結果を確認するだけで編集不可」にする**のが現実的

## 5. 共通で発生する作業

### 5.1 新しい CSS 変数を整備

tools/*.html で使っている Tailwind 固有の `bg-sfc-dark`・`text-slate-*` を CSS 変数に置き換える:

```css
/* _tool-common.css に追加 */
:root {
  --sfc-dark:    #004d29;
  --sfc-primary: #006837;
  --sfc-light:   #e8f5e9;
  --sfc-border:  #a5d6a7;
}
@media (prefers-color-scheme: dark) {
  :root {
    --sfc-dark:    #0a6b3a;
    --sfc-primary: #10b981;
    --sfc-light:   #064e3b;
    --sfc-border:  #065f46;
  }
}
```

これを入れた時点で tools/ から **Tailwind CDN を外せる可能性**（作業Hの回答)。
3 ツールが全部ネイティブ化されれば、inspection.html の Tailwind だけ残るが、それも独立ファイルなのでビルド不要。

### 5.2 共通トースト関数の追加

`_tool-common.js` に `showToast(msg, type)` を共通関数として追加。現状 `inspection.html` 内に独自実装があるので、それを抽出して共通化。

### 5.3 フォームバリデーション共通化

- 必須チェック
- 日付順序チェック（着工 < 建て方 < 引渡し）
- 物件名の重複チェック（既存確認）

## 6. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| GAS 側の改修で既存 iframe が動かなくなる | 業務停止 | 既存の doGet は下位互換で残す。iframe 版は残しておき、ネイティブ版の安定後に削除 |
| 実データが壊れる（支払い誤登録、物件重複） | 業務データ汚損 | テスト物件で動作確認 → 1〜2 週間並行運用 → 安定後切替 |
| ネイティブ版の UI が iframe より使いにくい | ユーザー不満 | 既存 UI をそのまま模写する。大きな改善は後回し |
| ダークモードで入力フォームが読めない | 視認性 | 作業 B で入れたパターン（CSS 変数 + `@media prefers-color-scheme`）に従う |
| Notion API レート制限 | 一時エラー | 既存実装のエラーハンドリングをそのまま流用 |
| GAS の deployed URL 変更忘れ | 叩けない | 既存の GAS URL をそのまま使う（変えない） |

**この草案について**

この設計は「こう進めれば安全」という紙の計画であって、コードはまだ1行も書いていない。着手前にもう一度読み返し、必要なら修正してから進めるのを推奨。
